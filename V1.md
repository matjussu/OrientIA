# Rapport d'analyse — OrientIA (INRIA AI Grand Challenge)

## 1. Synthèse du projet

**Thèse centrale** : un RAG spécialisé avec re-ranking par labels institutionnels produit des réponses d'orientation supérieures aux LLM généralistes sur 6 critères mesurables, avec le même modèle de base.

**Livrable** : un outil web (React + FastAPI + Mistral + FAISS) démontré par un benchmark comparatif contre ChatGPT et Mistral brut sur 30 questions, le tout livré en **4 semaines**.

**Positionnement** : c'est un projet de **recherche appliquée**, pas un produit. La valeur du rendu tient autant à la méthodologie de benchmark qu'au code lui-même.

---

## 2. Analyse des choix techniques

`★ Insight ─────────────────────────────────────`
- **RAG seul vs fine-tuning** : le plan cite arXiv 2401.08406 et 2403.01432 — ces papiers montrent que pour des données factuelles qui changent (Parcoursup annuel), le RAG est le bon outil. Le fine-tuning hallucinerait les chiffres, ce qui casserait la thèse "réalisme sourcé".
- **Re-ranking par labels = l'innovation** : mathématiquement triviale (multiplier le score cosinus par 1.5 si SecNumEdu), mais c'est précisément ce qui est publiable — une intervention minimale, reproductible, mesurable. C'est l'équivalent d'un "A/B test scientifique".
- **FAISS CPU vs Mistral Agents API** : le plan tranche pour FAISS maison. C'est le bon appel *scientifiquement* — l'Agents API est une boîte noire, on ne pourrait pas intervenir sur le re-ranking, donc pas de contribution mesurable.
`─────────────────────────────────────────────────`

### Stack — évaluation

| Choix | Pertinence | Commentaire |
|---|---|---|
| **Mistral API** | OK Solide | Free tier 1Md tokens/mois, souveraineté, argument INRIA parfait |
| **FAISS-CPU** | OK Solide | 200-300 fiches tiennent en RAM en quelques Mo, pas besoin de GPU ni de DB vectorielle |
| **mistral-embed** | A vérifier | Dimension et pricing à confirmer — risque d'écart avec la doc citée |
| **FastAPI + React + Vercel** | OK Standard | Déploiement en quelques heures, 0€ |
| **LLM-as-a-Judge (Claude/GPT-4)** | Nuance | Le plan note correctement d'éviter Mistral juge (self-preference bias), mais oublie que Claude/GPT-4 ont leur propre biais pro-US/pro-privé. À noter dans les limites du benchmark. |

---

## 3. Forces du plan

1. **Scope maîtrisé** : 200-300 fiches, 4 domaines. Suffisant pour la preuve de concept, pas de fuite de scope vers "couvrir les 14 000 formations".
2. **Thèse falsifiable** : le benchmark chiffre l'écart sur 6 critères. Si le re-ranking n'améliore rien, ça se verra — c'est de la vraie démarche scientifique.
3. **Argument politique aligné sur le jury INRIA** : souveraineté, AI Act, conformité RGPD, impact social chiffré (8 000€/an par mauvaise orientation).
4. **Données 100% publiques et réutilisables** : aucune dépendance à un scraping fragile ou à une API payante.
5. **Sources de vérité officielles** : Parcoursup + ANSSI + ONISEP + ROME → argument "nous ne pouvons pas être contestés sur les chiffres".
6. **System prompt bien construit** : structure en 4 couches (identité, comportement, format, garde-fous), règles actionnables, pas de persona vague.
7. **Plan d'exécution réaliste** : 1 semaine par étape, livrables concrets par semaine.

## 4. Faiblesses et risques identifiés

### Risques techniques

| # | Risque | Gravité | Mitigation possible |
|---|---|---|---|
| R1 | **ONISEP API** : limite 5 req/jour sans auth, pipeline JWT non testé | Moyenne | Commencer par le bulk CSV depuis data.gouv.fr, API en backup |
| R2 | **Qualité des embeddings sémantiques français** : `mistral-embed` est bon mais moins benchmarké que OpenAI/BGE sur le français | Moyenne | Prévoir un test A/B rapide vs un BGE-m3 local en fallback |
| R3 | **Re-ranking par labels = solution hard-codée** : facteurs 1.5, 1.3, 1.1 choisis arbitrairement | Moyenne | Ajouter une section "sensibilité au choix des poids" dans le benchmark (grid search sur 3-5 valeurs) |
| R4 | **SecNumEdu structuré "manuellement ~1h"** : sous-estimation probable, la liste change, pas d'export structuré officiel | Faible | Accepter un CSV figé à une date, documenter le snapshot |
| R5 | **30 questions = petit échantillon** : pas de significativité statistique | Moyenne | Assumer le cadre "étude exploratoire", présenter les IC 95% avec bootstrap |

### Zones floues dans le plan

1. **Taux d'accès Parcoursup** : le CSV Parcoursup public donne les taux d'admission, mais le mapping formation-Parcoursup ↔ formation-ONISEP n'est **pas trivial**. Le plan ne décrit pas cette jointure, c'est probablement le vrai travail difficile de la semaine 1.
2. **Chunking** : le plan embed 1 fiche = 1 chunk. Pour des fiches courtes ça marche, mais si une fiche fait 2000 tokens, la pertinence chute. À trancher dès la S1.
3. **Formations "non labellisées pour comparaison"** : nécessaires pour mesurer le boost, mais le critère de sélection n'est pas défini. Il faut un échantillonnage explicite (ex : top-20 EPITA, Epitech, Guardia, Ionis…) pour que le benchmark démontre bien la correction du biais.
4. **ChatGPT "réponses pré-enregistrées"** : à quelle date ? Dans quel contexte ? Le plan reconnaît la contrainte budget mais n'explicite pas la méthodologie reproductible.
5. **Benchmark Mistral brut vs Mistral+RAG** : c'est l'ablation la plus importante scientifiquement (isole l'effet RAG), mais le plan ne détaille pas si Mistral brut reçoit le même system prompt sans contexte, ou un prompt par défaut.

### Risques méthodologiques

- **Biais du juge** : même en choisissant Claude, la grille pousse vers "formation publique labellisée = bonne réponse". Un juge qui lit la rubrique **sait** implicitement quel système a été conçu pour bien scorer. **Mitigation obligatoire** : double-aveugle — le juge ne doit pas savoir quelle réponse vient de quel système, et les labels "système A/B/C" doivent être randomisés par question.
- **Cherry-picking** : les 3 démos vidéo (cyber, HEC, géopolitique) sont **celles où le RAG gagne forcément**. Pour le dossier INRIA, prévoir aussi 1-2 cas où le RAG **n'améliore pas**, ça renforce la crédibilité scientifique.

## 5. Points forts du system prompt (couche 2)

`★ Insight ─────────────────────────────────────`
- Les 4 règles (neutralité/réalisme/agentivité/sourçage) sont **parfaitement alignées** avec les 6 critères du benchmark — le prompt a littéralement été conçu pour maximiser la grille. C'est intellectuellement honnête à condition de le dire : "nous définissons la qualité, puis nous mesurons sur cette définition."
- La seuils de sélectivité (10/30/60%) sont du **binning arbitraire**. Risque que le juge note différemment selon son propre cadrage. À documenter comme "choix explicite, pas donné du monde".
`─────────────────────────────────────────────────`

## 6. Questions à clarifier avant de commencer

Voici les 8 points à verrouiller avant de taper une ligne de code :

1. **Mapping Parcoursup ↔ ONISEP** : existe-t-il un identifiant commun (RNCP ? nom d'établissement ?), ou faut-il un matching fuzzy ? C'est le chemin critique de la S1.
2. **Scope final** : confirme-t-on les 4 domaines (cyber, data/IA, commerce, droit) ou on en fait 2 en profondeur ?
3. **Langue du benchmark** : français uniquement ? (important pour le choix d'embeddings et le juge)
4. **Juge** : Claude ou GPT-4 ? Si budget permet, faire les deux et moyenner.
5. **Coût réel** : le free tier Mistral suffit-il pour générer ~30 questions × 3 systèmes × 2-3 runs + embeddings de 300 fiches ? (oui probablement, mais à chiffrer)
6. **Date de soumission INRIA** : quand exactement ? Les 4 semaines partent de quand ? Ça conditionne toute la priorisation.
7. **Open source** : repo public dès J1 ou à la soumission ? (argument auditabilité du dossier)
8. **Tu veux un frontend personnalisé ou un template React minimal suffit ?** — le vrai impact du dossier est sur le benchmark, pas sur le polish UI.

## 7. Recommandation d'ordre d'attaque

Le plan tel quel est bon, mais je réordonnerais légèrement pour **dérisquer** :

| Ordre | Tâche | Durée | Raison |
|---|---|---|---|
| **0** | Créer le repo, configurer les clés API Mistral, télécharger les CSV bruts | 0.5j | Valider les accès avant de s'engager |
| **1** | Jeu d'évaluation (30 questions + réponses idéales) d'abord | 1j | Définit le "succès" avant d'optimiser le système |
| **2** | Collecte + mapping Parcoursup↔ONISEP sur **1 domaine seulement** (cyber) | 2-3j | Valide la pipeline de jointure, le plus risqué |
| **3** | Pipeline RAG end-to-end sur les 80 fiches cyber | 2j | Premier MVP testable, même médiocre |
| **4** | Premier benchmark sur les 5 questions cyber → feedback | 1j | **Go/no-go** sur la thèse avant d'élargir |
| **5** | Si feedback positif : étendre aux 3 autres domaines | 3-4j | Montée en charge mécanique |
| **6** | Benchmark complet + analyse | 2j | |
| **7** | Interface + vidéo démo | 3j | En dernier, le moins différenciant scientifiquement |

**Principe** : tu veux savoir **dès la fin de la S1** si le re-ranking produit un effet mesurable. Si non, le plan entier doit pivoter. Aller "large puis benchmarker" risque 3 semaines perdues.

---

## 8. Verdict global

**Le plan est publiable et exécutable en 4 semaines.** C'est un vrai projet de recherche, pas un hackathon déguisé. Les choix techniques sont défendables, la thèse est falsifiable, le scope est réaliste, l'argument politique est aligné sur l'INRIA.

**Les 3 points critiques à traiter avant d'implémenter** :
1. Le mapping des datasets (vrai travail caché de la S1).
2. Le protocole en double-aveugle du juge LLM (sinon le benchmark est contestable).
3. L'ablation Mistral brut vs Mistral+RAG bien définie (sinon la contribution du RAG n'est pas isolée).
